# Code signing & notarization — macOS

This document covers what's required to ship a fully signed, **notarized**
Journal build for macOS. Until a Developer ID certificate is configured,
builds are **ad-hoc signed but not notarized** — they launch, but users see a
one-time Gatekeeper prompt (see "End-user workaround" below).

## State today

`../electron-builder.yml` sets `mac.identity: "-"`, which makes electron-builder
**ad-hoc sign** the whole bundle (`codesign --sign -`) at packaging time. This
is required: arm64 binaries on macOS 11+ will not launch without *any*
signature, and the deep ad-hoc sign covers the Electron framework and helper
processes that macOS would otherwise kill silently.

Ad-hoc signing is **not** the same as Developer ID signing + notarization:

- **Ad-hoc (today):** the app launches, but macOS shows a "cannot verify"
  prompt on first open. The user clicks through it once (System Settings →
  Privacy & Security → Open Anyway).
- **Developer ID + notarized (this doc):** no prompt at all. Required for
  frictionless distribution to non-technical users.

The release workflow at `.github/workflows/release-desktop.yml` and the
electron-builder config already read the standard signing env vars. They're
empty today; when you add the secrets below, electron-builder upgrades the
ad-hoc sign to a full Developer ID sign + notarization automatically.

## What you need

1. **Apple Developer Program membership** ($99/yr). Sign up at
   <https://developer.apple.com/programs/>.

2. **Developer ID Application certificate** (NOT Mac App Store). Two options:

   a. From the Apple Developer console:
      - Certificates, IDs & Profiles → Certificates → "+"
      - Choose "Developer ID Application"
      - Upload a CSR generated by Keychain Access on a Mac you control
      - Download the `.cer`, double-click to install into your login keychain
      - Export both the cert and its private key as a single `.p12` file
        (Keychain Access → right-click cert → Export, set a password)

   b. Or use `xcrun` to generate one if you've shipped via Xcode before.

3. **App-specific password** for notarization:
   - <https://appleid.apple.com> → Sign-In and Security → App-Specific Passwords
   - Label it something like "electron-builder-journal"
   - Copy the 16-character password — you can't retrieve it later

4. **Team ID** — 10-character string. Find it at
   <https://developer.apple.com/account> → Membership Details.

## GitHub Actions secrets

Once you have the above, add these to the repo as Actions secrets at
`https://github.com/DiamondDime/trading-journal/settings/secrets/actions`:

| Secret name                   | Value                                                     |
| ----------------------------- | --------------------------------------------------------- |
| `CSC_LINK`                    | base64-encoded `.p12` (run `base64 -i cert.p12 \| pbcopy`) |
| `CSC_KEY_PASSWORD`            | The password you set when exporting the `.p12`            |
| `APPLE_ID`                    | Your Apple Developer email (`you@example.com`)            |
| `APPLE_APP_SPECIFIC_PASSWORD` | The 16-char app-specific password                         |
| `APPLE_TEAM_ID`               | Your 10-char Team ID                                      |

The workflow passes these through to electron-builder automatically. Next
push of a `v*.*.*` tag will produce a signed + notarized build.

## Local signing (optional, for testing before pushing a tag)

```bash
export CSC_LINK=/path/to/cert.p12
export CSC_KEY_PASSWORD='your-cert-password'
export APPLE_ID='you@example.com'
export APPLE_APP_SPECIFIC_PASSWORD='xxxx-xxxx-xxxx-xxxx'
export APPLE_TEAM_ID='ABCDEFGHIJ'

pnpm electron:build
```

The first run will take several minutes — notarization makes a round-trip to
Apple. You'll see `notarization successful` near the end of the output.

## Verifying a build

After installing the resulting `.dmg`:

```bash
# Cert chain / signing details
codesign -dv --verbose=4 /Applications/Journal.app

# Gatekeeper assessment (should print "accepted")
spctl --assess --verbose=4 /Applications/Journal.app
```

You're looking for:
- `Authority=Developer ID Application: <your name> (<team id>)`
- `accepted source=Notarized Developer ID`

If `spctl` prints `rejected`, the build is signed but not notarized — check
the `APPLE_*` env vars.

## End-user workaround (until certs are wired)

Unsigned builds trigger this Gatekeeper warning on first launch:

> "Journal" can't be opened because Apple cannot check it for malicious
> software.

Workaround:

1. Open Finder, navigate to `/Applications`
2. **Right-click** `Journal.app` → **Open**
3. Click **Open** in the second dialog

After the first launch, macOS remembers the user's choice and future launches
go through normally. Document this in the README's install section.
